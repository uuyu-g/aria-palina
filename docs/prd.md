# 📄 Product Requirements Document (PRD): aria-palina Ecosystem

> **Status:** Draft
> **Last Updated:** 2026-04-11

## 1. プロダクトビジョンと背景 (Vision & Background)

### 1.1 背景 (The Problem)

現代のWeb開発において、フロントエンドエンジニアの多くはMac環境（VoiceOver）で開発を行いますが、実際の視覚障害者ユーザーやQAエンジニアの多くはWindows環境（NVDA）を使用しています。

加えて、CSS（TailwindやGrid等）の表現力が飛躍的に向上した結果、**「視覚的には完璧に整理されているが、HTMLの構造（セマンティクス）は完全に破綻している」** というアンチパターンが蔓延しています。

これにより、「開発者が目視で確認しヨシとしたものが、スクリーンリーダーでは全く使い物にならない」という問題が多発し、QAフェーズでの深刻な手戻り（フィードバックループの遅延）を引き起こしています。

### 1.2 プロダクトビジョン (The Vision)

**"Visual noise off. Semantics on."**

aria-palinaは、Webページの「視覚的な化粧（CSS）」を完全に剥ぎ取り、マシンと支援技術が解釈する真の構造である **「アクセシビリティツリー（AOM: Accessibility Object Model）」** を、開発者のターミナルやDevTools上に直接描き出します。

開発者がGUIのノイズに騙されることなく、コードのセマンティクスと論理的なナビゲーション順序のみに集中できる「究極のアクセシビリティ開発環境」を提供します。

## 2. ターゲットユーザー (Target Audience)

1. **フロントエンドエンジニア (主ターゲット):** Mac環境で開発を行い、ターミナルやエディタから極力離れずに、高速にアクセシビリティのデバッグを行いたい層。
2. **QAエンジニア:** リリース前に論理的なDOM構造や読み上げ順序が仕様（ARIA-AT等）に準拠しているかをテストする層。
3. **アクセシビリティ・スペシャリスト:** 複雑なUIコンポーネントのアクセシビリティ・ガイドラインを策定・検証する層。

## 3. コア・ユースケースと解決する課題 (Core Use Cases)

このツールは、以下のような「視覚の呪い」によるアンチパターンを開発者に突きつけ、修正への明確な道筋を示します。

1. **CSSによる視覚表現の罠を暴く:**
   - `<div>` の羅列で作られた偽物のテーブルを「ただのテキストの連続」として可視化し、正しい `<table>` タグへの修正を促す。
   - 見た目だけを大きくした偽物の見出しを検知し、ドキュメントのアウトライン崩壊を防ぐ。

2. **隠蔽とフォーカストラップの検証:**
   - モーダルを開いた際、視覚的には黒いオーバーレイで隠れている背面の要素が、ツリー上に「読める状態」で残ってしまっている（フォーカス抜け）ことを即座に検知する。

3. **状態変化（State）の沈黙の打破:**
   - SPAの画面遷移時や、ドラワー（モードレスUI）が出現した際に、フォーカス移動や適切なアナウンス（`aria-live` 等）が行われていない「沈黙の状態」を可視化する。
   - CSSの `pointer-events: none` で無効化したつもりの「幽霊ボタン」にキーボードフォーカスが当たることを証明する。

## 4. 機能要件 (Functional Requirements)

aria-palinaは、単一のツールではなく、開発のあらゆるフェーズに対応する「エコシステム」として以下の機能を提供します。

### 4.1 CLI モード (One-shot Validation)

- 指定したURL（ローカル/リモート）のアクセシビリティツリーを抽出し、標準出力（stdout）に書き出す。
- **スマートTTY判定:**
  - 人間がターミナルで読む場合（TTY）は、インデントとカラーを付与し、DOMの階層構造を視覚的に表現する。
  - パイプライン（`grep`, `jq` 等）に渡される場合は、インデントとカラーを排除したプレーンなフラット配列を出力する。
- フォーマットはプレーンテキスト（NVDAライクな読み上げ文字列）と、構造化されたJSONをサポートする。

### 4.2 TUI モード (Immersive Debugging)

- ターミナル上で動作するインタラクティブなUI。
- **デュアルナビゲーション:**
  - ブラウズモード（矢印キー）: DOMの順序通りに全ノードを移動。
  - フォーカスモード（`Tab` / `Shift+Tab`）: インタラクティブ要素のみをジャンプ。
- **クイックジャンプ:** 見出し（`H` キー）、ランドマーク（`D` キー）間の移動。
- **Matrix View (Headed同期モード):** `--headed` オプション起動時、TUI上でフォーカスした要素を、並べて表示したブラウザ上で視覚的にハイライト（青い網掛け等）する。TUI → ブラウザの片方向同期のみを提供する。

### 4.3 DevTools Extension モード

- Chrome拡張機能として、DevToolsのパネル内にTUIと同等のツリービューを提供する。
- 開発者がブラウザで画面を操作しながら、横のパネルでリアルタイムにツリーの変化を確認できる。

### 4.4 BDD Test Utilities (Automated Specs)

- TUIで確認した「正しい読み上げ順序」を、CucumberやVitest等の自動テストとして記述できるアサーションライブラリを提供する（例: `expect(page).toHavePalinaTextSequence(...)`）。

## 5. 非機能要件 (Non-Functional Requirements)

- **パフォーマンス:** 数千ノードを持つ巨大なWebページであっても、TUI/DevTools上で遅延なく（60fps相当で）キーボードナビゲーションが可能であること。全ノードの同時レンダリングを避け、仮想スクロール（Windowing）を必須とする。
- **ポータビリティ:** Windows/Mac/Linux の Node.js 環境で動作すること。
- **パイプライン耐性:** CLI実行時、プログレスログやエラーは必ず `stderr` に出力し、`stdout` には純粋なデータのみを流すことで、Unixパイプライン処理を絶対に破壊しないこと。
- **拡張性:** ブラウザ自動化ツール（Playwright, Puppeteer）や通信プロトコル（CDP）の実装にコアロジックが密結合しないよう、DI（依存性の注入）パターンを用いて設計すること。

## 6. 成功指標 (Success Metrics)

- 導入プロジェクトにおける、QAフェーズでのアクセシビリティ関連バグ（特に読み上げ順序やセマンティクスに関するもの）の指摘数の大幅な削減。
- 開発者がアクセシビリティのデバッグにかける時間の短縮。
