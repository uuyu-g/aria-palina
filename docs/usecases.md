# 🎭 UX Simulations & Use Cases (aria-palina 道場)

> **「目は騙せても、アクセシビリティツリーは騙せない」**

> **Status:** Draft
> **Last Updated:** 2026-04-11
> **Related:** [PRD](./prd.md) / [DD](./dd.md)

このドキュメントは、現代の Web 開発で頻発する「視覚的なアンチパターン」を、aria-palina がどのように暴き出し、開発者にどのような「アハ体験（気づき）」をもたらすかをまとめたシミュレーション集です。

## カテゴリ1: 構造とセマンティクスの崩壊 (Structure & Semantics)

CSS フレームワークの発展により、HTML タグの意味（セマンティクス）を無視しても美しいレイアウトが作れるようになりました。しかし、それはスクリーンリーダーにとって「骨抜きにされた文書」を意味します。

### 1.1 偽物のテーブル（`display: grid` の罠）

**🚨 アンチパターン:** Tailwind CSS や Grid レイアウトを使い、全て `<div>` でユーザー一覧表を実装した。

**👁️ ユーザーの絶望:** スクリーンリーダーで読むと、列の文脈が完全に消失し「田中」「管理者」「鈴木」「一般」という単語が無限に縦に並ぶだけになる。自分が今何のデータを読んでいるのか絶対に迷子になる。

- **TUI での検証 (Before):**

  ```text
  [テキスト] 田中太郎
  [テキスト] tanaka@example.com
  [テキスト] 管理者
  [ボタン] 編集
  ```

- **💡 気づきと修正:** `<table>`, `<th>`, `<td>` などの正しい表組みタグを使用する。
- **TUI での検証 (After):**

  ```text
  > [セル 3/4, 権限] 管理者
  ```

  *結果：TUI が「自分が今どの列にいるか」のコンテキストを自動補完して表示するようになり、完璧なテーブルナビゲーションが実現した！*

### 1.2 偽物の見出し（巨大なテキストの罠）

**🚨 アンチパターン:** LP（ランディングページ）の各セクションタイトルを `<div class="text-3xl font-bold">` で実装した。

**👁️ ユーザーの絶望:** 長いページを飛ばし読みするための「見出しジャンプ機能」が一切使えず、ページの先頭から一言一句読まないと全体像が掴めない。

- **TUI での検証 (Before):** 開発者が TUI 上で `H` キー（見出しジャンプ）を押す。

  ```text
  [システム通知] 次の見出しが見つかりません。
  ```

- **💡 気づきと修正:** 意味構造に合わせて `<h2>` や `<h3>` を使用する。
- **TUI での検証 (After):** `H` キーを押すたびにカーソルが爆速で移動する。

  ```text
  > [見出し2] 特徴
  > [見出し2] 料金体系
  ```

## カテゴリ2: フォーカスと画面遷移の迷宮 (Focus & Routing)

リッチな SPA（Single Page Application）やダイアログは、フォーカス管理を怠ると視覚障害者を「出口のない迷路」に閉じ込めてしまいます。

### 2.1 モーダルダイアログのフォーカス裏抜け

**🚨 アンチパターン:** 退会モーダルを開いた際、背景を黒い半透明の `div` で覆っただけで、背景要素の非表示化やフォーカストラップを実装していない。

**👁️ ユーザーの絶望:** モーダル内で Tab キーを押していると、突然「プライバシーポリシー」など、視覚的には見えないはずの背面ページ（フッター等）にフォーカスが飛び、現在地を見失う。

- **TUI での検証 (Before):** TUI のフォーカスモード（Tab キー）でモーダル内を進む。

  ```text
    [ボタン] 退会する
  > [リンク] フッター: プライバシーポリシー  <-- モーダルの外に突き抜けた！
  ```

- **💡 気づきと修正:** 背景コンテナに `aria-hidden="true"` を付与し、JS でモーダル内の最初と最後の要素間でフォーカスをループさせる。
- **TUI での検証 (After):** ツリーから背景要素が完全に消え去り、モーダル内のボタンだけでカーソルがループする。

### 2.2 SPA 遷移の「沈黙」

**🚨 アンチパターン:** React Router などで画面遷移した際、DOM を書き換えただけでフォーカスを移動させていない。

**👁️ ユーザーの絶望:** リンクを押してページが切り替わっても何もアナウンスされず、「ボタンが壊れているのか？」「ページが変わったのか？」が全くわからない。

- **TUI での検証 (Before):** TUI 上でリンクを Enter で叩いた後、ツリーは更新されるが、カーソル位置が元の場所（虚空）に取り残される。
- **💡 気づきと修正:** 遷移完了時、ルーターのイベントをフックして新しいページの `<h1>` にフォーカスを移す。
- **TUI での検証 (After):** リンクを叩いた瞬間、TUI のカーソルが自動的に上部へジャンプする。

  ```text
  > [見出し1] 商品Aの詳細  <-- ここに自動ジャンプし、読み上げられる！
  ```

> **🛠️ Tool Support:** `palina --tui` はライブ更新が既定で有効。`DOM.documentUpdated` /
> `Page.frameNavigated` を購読し、SPA 遷移やモーダル展開の完了後に AX ツリーを
> 自動で取り直す。`backendNodeId` でカーソルを追従するため、見出しが同じノードの
> まま残る場合は位置が保持される。自動追従を止めたい場合は `--no-live`、
> 任意のタイミングで取り直すには `r` キー、ライブのトグルは `L` キー。
> ワンショット CLI では `--wait-for-selector '#page-ready'` / `--wait-for-function`
> / `--delay 1500` で「落ち着いた」状態を待ってから抽出できる。
> NVDA 風の aria-live 通知 (`role="status"` / `role="alert"` 等のテキスト変化) は
> TUI のステータスバーに `♪` (polite) / `!` (assertive) 付きで表示される。

### 2.3 ドラワー UI の「沈黙の出現」

**🚨 アンチパターン:** サイドメニュー（ドラワー）を開くボタンを押しても、視覚的にスライドインしてくるだけで、フォーカスはトリガーボタンに残ったまま。

**👁️ ユーザーの絶望:** メニューが開いたことに気づけない。メニューの中身を読むために、ページ全体を再度 Tab キーで舐め回す必要がある。

- **TUI での検証 (Before):** Enter を押しても `[ボタン] メニュー` に留まったままになる。
- **💡 気づきと修正:** トリガーに `aria-expanded` と `aria-controls` を設定し、開いた瞬間にドラワー内の最初のフォーカス可能要素（閉じるボタン等）へフォーカスを移す。
- **TUI での検証 (After):**

  ```text
  > [ダイアログ] ナビゲーションメニュー
      [ボタン] 閉じる  <-- 自動的にここにジャンプ！
  ```

## カテゴリ3: 状態とインタラクティビティ (State & Interactivity)

「ボタンが押せるのか」「メニューが開いているのか」。視覚的な変化（色やアイコン）だけで状態を伝えている UI は、スクリーンリーダーには「沈黙の UI」となります。

### 3.1 独自アコーディオンの沈黙

**🚨 アンチパターン:** FAQ のアコーディオンを `<div>` と `onClick` で実装し、開閉状態をアイコンの向き（CSS）だけで表現している。

**👁️ ユーザーの絶望:** その要素がボタンであることも、展開できることも、現在開いているのか閉じているのかも全くわからない。

- **TUI での検証 (Before):**

  ```text
  > [テキスト] サービスは無料で使えますか？  <-- ボタンとして認識されない
  ```

- **💡 気づきと修正:** `<button>` を使用し、`aria-expanded="true/false"` をステートと連動させる。
- **TUI での検証 (After):**

  ```text
  > [ボタン] サービスは無料で使えますか？ (折りたたみ)
  ```

  *Enter キーを押すと即座に `(展開)` に切り替わり、ステートがリアルタイムに伝わる！*

### 3.2 CSS で無効化した幽霊ボタン

**🚨 アンチパターン:** 入力必須のフォームで、送信ボタンを `pointer-events: none; opacity: 0.5;` で「見た目だけ」無効化している。

**👁️ ユーザーの絶望:** スクリーンリーダー上は通常のボタンとして存在するため、押せると思って Enter キーを何度も叩くが何も起こらない（幽霊ボタン）。

- **TUI での検証 (Before):**

  ```text
  > [ボタン] 送信  <-- 普通のボタンとしてフォーカスが当たる
  ```

- **💡 気づきと修正:** HTML の `disabled` 属性を使用する、または `aria-disabled="true"` を設定する。
- **TUI での検証 (After):**

  ```text
  > [ボタン] 送信 (利用不可)
  ```

### 3.3 リッチなドロップダウンメニューの罠

**🚨 アンチパターン:** ホバーで開く巨大なメガメニューを `<ul>` と `<li>` だけで実装。

**👁️ ユーザーの絶望:** メニューが開いた際、それが「親メニューと紐付いたサブメニュー」であることが伝わらず、ただ唐突に膨大なリストが出現したように感じる。

- **TUI での検証 (Before):**

  ```text
  > [リスト]  <-- メニューとしてのセマンティクスがない
      [リスト項目] 家電
  ```

- **💡 気づきと修正:** 全体に `role="menu"`、各項目に `role="menuitem"` を付与する。
- **TUI での検証 (After):**

  ```text
  > [メニューボタン] 製品 (展開)
      [メニューアイテム 1/3] 家電
  ```

## カテゴリ4: メディアと代替テキスト (Media & Labels)

### 4.1 画像の `alt` の迷宮（装飾 vs 情報）

**🚨 アンチパターン:** 記事のアイキャッチ画像（装飾用）に `alt="アイキャッチ"` と入れ、重要な売上グラフの画像に `alt="グラフ1"` と入れている。

**👁️ ユーザーの絶望:** 不要な「アイキャッチ」という音声で時間を奪われ、本当に必要な「売上の推移」という情報が全く手に入らない。

- **TUI での検証 (Before):**

  ```text
  > [画像] アイキャッチ画像  <-- ノイズ
  > [画像] グラフ1           <-- 情報不足
  ```

- **💡 気づきと修正:** 装飾画像は `alt=""` でツリーから隠し、グラフには具体的な要約を記述する。
- **TUI での検証 (After):**

  ```text
  > [画像] 2023年の売上推移グラフ、右肩上がり
  ```

  *(装飾画像は TUI 上から完全に消滅し、クリーンなツリーになる)*

### 4.2 アイコンのみのボタン

**🚨 アンチパターン:** SNS シェアボタンを SVG アイコンのみで実装し、テキストラベルがない。

**👁️ ユーザーの絶望:** 「ボタン」「ボタン」「ボタン」と読み上げられるだけのロシアンルーレット状態。

- **TUI での検証 (Before):**

  ```text
  > [ボタン] (名前なし)
  ```

- **💡 気づきと修正:** `<button aria-label="Xでシェア">` などを付与する。
- **TUI での検証 (After):**

  ```text
  > [ボタン] Xでシェア
  ```

### 4.3 沈黙のエラーメッセージ

**🚨 アンチパターン:** フォームのバリデーションエラーを赤い文字で表示しているが、入力欄とのプログラム的な紐付けがない。

**👁️ ユーザーの絶望:** 入力欄にフォーカスを当てても「無効な入力です」という通知がなく、エラーの理由を探すためにページ内をさまよう必要がある。

- **TUI での検証 (Before):**

  ```text
  > [テキスト入力] メールアドレス
    [テキスト] メールアドレスは必須です  <-- 分離してしまっている
  ```

- **💡 気づきと修正:** 入力欄に `aria-invalid="true"` を設定し、`aria-describedby` でエラーメッセージの ID と紐付ける。
- **TUI での検証 (After):**

  ```text
  > [テキスト入力] メールアドレス (無効) - メールアドレスは必須です
  ```

  *入力欄にフォーカスした瞬間に、エラーの理由がセットで読み上げられる！*

## 🎯 BDD（振る舞い駆動開発）への応用

これらのシミュレーションは、すべて aria-palina の `@aria-palina/test-utils` を用いて、CI/CD で動作する自動テスト（Cucumber / Playwright）のシナリオとしてそのまま落とし込むことができます。

**例: モーダルのフォーカストラップの自動テスト**

```typescript
test('退会モーダルが開いたとき、背景がツリーから隠蔽されること', async ({ page }) => {
  await page.click('button:has-text("退会する")');

  // aria-palina Core を利用したカスタムマッチャー
  await expect(page).not.toHavePalinaText('[リンク] プライバシーポリシー');
  await expect(page).toHavePalinaTextSequence([
    '[ダイアログ] 退会確認',
    '[ボタン] キャンセル',
    '[ボタン] 退会する'
  ]);
});
```
